# RemindMe: Напоминания (ядро)

## Current State

Проект greenfield: исходного кода нет (`goga schema` → `[]`), ячеек нет. Заданы только инженерные
конвенции (`.goga/usages/conventions.md`: Python 3.10+, pydantic v2, относительные импорты,
Google-docstrings, pytest/ruff, структурированное логирование). Нет бота, моделей данных,
хранилища, планировщика.

## Description

Реализовать ядро **RemindMe** — многопользовательского Telegram-бота (aiogram 3.x, long-polling)
для управления напоминаниями. Бот создаёт фундамент приложения: скелет бота, модели данных,
асинхронное хранилище (SQLite) и планировщик срабатывания (APScheduler).

Каждый Telegram-пользователь изолирован по `user_id`. Время срабатывания хранится в UTC,
отображается и вводится в часовом поясе пользователя (TZ на пользователя, по умолчанию — TZ бота).

## Scope

**In scope:**
- Скелет aiogram-бота: `Bot`, `Dispatcher`, композиция `Router`, middleware ошибок, запуск
  long-polling, конфиг из `.env` через pydantic-settings
- Модели данных (pydantic v2, `kw_only=True`): пользователь (`telegram_id`, `timezone`),
  напоминание (`id`, `user_id`, текст, `trigger_at` UTC, статус)
- Хранилище: async SQLAlchemy 2.x + aiosqlite, engine/session-фабрика, репозитории
  напоминаний и пользователей
- Планировщик: APScheduler `AsyncIOScheduler`, разовые джобы с TZ-aware `run_date`,
  восстановление джоб из БД при старте
- Сценарии бота:
  - `/start` — онбординг, дефолтный TZ
  - задать часовой пояс (команда, по умолчанию TZ бота)
  - создать напоминание (FSM: текст + дата/время в TZ пользователя → расчёт UTC);
    ввод времени — абсолютный формат `YYYY-MM-DD HH:MM` (24ч, в TZ пользователя) и
    относительные формы (`завтра 15:00`, `через 2ч`, `через 30мин`, `через 1д`)
  - получить уведомление в назначенный момент (отправка сообщения)
  - просмотреть список напоминаний
  - удалить напоминание

**Out of scope:**
- Заметки и todo — отдельные задачи
- Рекуррентные (повторяющиеся) напоминания — только разовые
- Webhooks — только long-polling
- Админ-панель, аналитика
- Миграции через Alembic — на старте `create_all()`

## Acceptance Criteria

- Бот запускается long-polling, отвечает на `/start`
- Пользователь может задать TZ; время, введённое в абсолютном формате `YYYY-MM-DD HH:MM`
  или относительной форме (`завтра 15:00`, `через 2ч` и т.п.), корректно конвертируется в UTC
  (и обратно — для отображения в TZ пользователя)
- Созданное напоминание сохраняется и срабатывает в указанный момент (допуск — секунды),
  пользователь получает сообщение
- Список показывает только напоминания текущего пользователя (изоляция по `user_id`)
- Напоминание можно удалить
- После рестарта процесса запланированные напоминания восстанавливаются из БД
- Юнит-тесты покрывают конвертацию TZ и логику репозиториев; `ruff check` чист;
  `pytest tests/ -x` зелёный

## Risks and Constraints

- TZ-конвертация — основной источник ошибок: использовать `zoneinfo`, хранить только UTC,
  тестировать границы (переход на летнее время, неизвестный TZ)
- Планировщик и БД работают в одном event-loop aiogram — корректный порядок старта/останова
- SQLite — один writer: обработчики и планировщик пишут в одну БД. Включить WAL-режим,
  держать транзакции короткими, при `database is locked` — retry; для профиля нагрузки
  личного бота этого достаточно
- Токен бота и конфиг — только через `.env`, никогда в логах (требование conventions)
- Python 3.10+; все модели pydantic с `kw_only=True`; относительные импорты внутри пакетов
